# 方案 A：OpenAPI → TypeScript 类型自动生成

## 一、目标

建立从后端 FastAPI OpenAPI schema 到前端 TypeScript 类型定义的自动化生成管线，从根本上消除前后端字段名/类型约定不一致的问题。

***

## 二、现状分析

### 2.1 后端 OpenAPI schema 质量

| 项目                  | 当前状态                                                                         |
| ------------------- | ---------------------------------------------------------------------------- |
| FastAPI 应用          | `FastAPI(title="Arche", version="0.1.0")`，默认开启 `/openapi.json`               |
| Auth 中间件            | 已放行 `/docs`、`/openapi.json`、`/redoc`                                         |
| `response_model` 使用 | **全部路由均未使用** → 响应类型在 OpenAPI schema 中为泛化 `{}`                                |
| Pydantic 请求模型       | 路由中完整定义（如 `CreateJobRequest`、`LoginRequest` 等）                               |
| 响应格式                | 统一返回 `{"code": "ok", "message": "...", "data": result}`，`data` 是 Python dict |

### 2.2 前端类型定义

| 项目   | 当前状态                                                                                      |
| ---- | ----------------------------------------------------------------------------------------- |
| 类型定义 | 11 个 API 模块文件中手工维护约 40+ 个接口                                                               |
| 已知问题 | 多次出现字段名与后端不匹配（`user_id` vs `author_id`、`status` vs `status_code`、`count` vs `post_count`） |
| 测试体系 | 零测试基础设施                                                                                   |

### 2.3 工具链

选用 **`openapi-typescript`**（当前最新 v7.x）：

* 纯 Node.js，无 Java 依赖

* 支持 OpenAPI 3.0/3.1

* 生成零运行时的 `.d.ts` 类型文件

* 支持远程 URL 加载 schema

* 社区广泛使用（Stripe、GitHub 等采用）

## 四、具体实现细节

### 4.1 安装

```bash
cd frontend
npm install --save-dev openapi-typescript
```

### 4.2 生成脚本 `scripts/generate-api-types.mjs`

```javascript
/**
 * 从后端 FastAPI 获取 OpenAPI schema，生成 TypeScript 类型。
 *
 * 使用方式：
 *   1. 确保后端已在 http://localhost:8000 运行
 *   2. npm run generate:api
 */
import { spawn } from "node:child_process";
import { writeFile } from "node:fs/promises";
import openapiTS from "openapi-typescript";

const BACKEND_URL = process.env.BACKEND_URL || "http://localhost:8000";
const OUTPUT = new URL("../src/services/api/generated.d.ts", import.meta.url);

async function fetchSchema() {
  const res = await fetch(`${BACKEND_URL}/openapi.json`);
  if (!res.ok) throw new Error(`Failed to fetch schema: ${res.statusText}`);
  return res.json();
}

async function generate() {
  console.log(`Fetching OpenAPI schema from ${BACKEND_URL}/openapi.json ...`);
  const schema = await fetchSchema();

  console.log("Generating TypeScript types ...");
  const output = await openapiTS(schema, {
    commentHeader: "/* eslint-disable */\n// Auto-generated by openapi-typescript. Do not edit.",
  });

  await writeFile(OUTPUT, output);
  console.log(`Types written to ${OUTPUT}`);
}

generate().catch((err) => {
  console.error(err);
  process.exit(1);
});
```

### 4.3 package.json 脚本

```json
{
  "scripts": {
    "generate:api": "node scripts/generate-api-types.mjs",
    "generate:api:check": "node scripts/generate-api-types.mjs && git diff --exit-code src/services/api/generated.d.ts"
  }
}
```

### 4.4 使用方式

```typescript
// 在 API 函数中使用生成的类型
import type { components } from './generated.d'

// 假设 OpenAPI schema 中有 components/schemas/BlogPost
type BlogPost = components["schemas"]["BlogPost"];

// 路由路径类型
import type { paths } from './generated.d'
type PostParams = paths["/api/blog/posts"]["get"]["parameters"];
```

## 五、风险与应对

| 风险                                    | 应对                                                                        |
| ------------------------------------- | ------------------------------------------------------------------------- |
| 后端无 `response_model`，生成的响应类型是泛化的 `{}` | Phase 1-2 先产出可用类型（请求体类型），Phase 4 逐步补齐响应模型                                 |
| 生成的 `.d.ts` 文件很大                      | openapiTS 只生成类型不生成代码，体积可控。可在脚本中过滤仅保留需要的 schemas                           |
| 后端变更后前端类型未及时重新生成                      | CI 中跑 `generate:api:check` 检测漂移；开发流程中养成 re-generate 习惯                    |
| 手工类型和生成类型共存期间的冲突                      | 生成的文件加 `/* eslint-disable */` 头部，命名上区分（`generated.d.ts` vs 手工 `types.ts`） |

***

## 六、验证步骤

1. 启动后端：`uv run uvicorn backend.main:app --reload`
2. 运行：`cd frontend && npm run generate:api`
3. 确认生成了 `src/services/api/generated.d.ts`
4. 打开文件验证：应有 `paths`、`components`、`operations` 等命名空间导出
5. 在 `blog.ts` 中找一处替换试用：`import type { components } from './generated'`
6. 运行 `npm run type-check` 确认无类型错误
7. 运行 `npm run lint` 确认通过
8. 运行 `git diff` 确认生成文件内容合理
