# Issue 管理工作流（纯决策模式）

## 场景触发条件
当需要处理 GitHub Issue（已知问题）时，且当前工作模式为**纯决策管理**（不涉及代码修改）。

## 工作流

### 第一步：Code Review 验证
对每个 Issue 进行代码审查，确认问题是否真实存在以及问题本质：

1. 拉取 Issue 的详细信息（标题、描述、标签）
2. 审查相关的后端代码（路由、服务、模型）
3. 审查相关的前端代码（组件、API 调用、路由配置）
4. 分析问题的根本原因

### 第二步：根据审查结果分类处理

#### 情况 A：问题不存在/已被修复
- 直接关闭 Issue
- 评论说明审查结论（为何问题不存在）
- 无需额外沟通确认

#### 情况 B：问题真实存在，但发现更深层次的新问题
- 在 Issue 下发表评论，补充完整的审查分析报告
- 报告中需包含根因分析、涉及文件、修复方向建议

#### 情况 C：复合问题（包含多个独立子问题）
- 将每个独立子问题拆分为新的 Issue
- 在新创建的 Issue 中标注关联的父 Issue
- 在父 Issue 下发表评论，列出所有子 Issue 的链接
- 父 Issue 本身不关闭，用于追踪整体进度

### 工具使用
- 用 `gh` CLI 操作 Issue（close、comment、create）
- 用 Code Review 工具审查代码
- 避免对代码库进行任何文件修改
