# 工程决策原则

> 本文件定义了在 Arche 项目中做技术决策时遵循的具体原则。
> 本文件受 `constitution.md`（宪法）约束，不得违背宪法精神。
>
> 宪法核心指引：**没有调查，就没有发言权** | **实践是检验真理的唯一标准**

## 借鸡生蛋原则

**核心思想**：拿现有东西借鸡生蛋，研究标准，实践最佳实践，而非盲目自研。
**宪法依据**：抄答案，不自研 | 从群众中来，到群众中去

- **基础设施用现成的**：Web 框架（FastAPI / Starlette）、数据库（SQLAlchemy / 原生 SQLite）、前端生态（Vue 3 + Vite）、构建部署等，已有大量成熟方案和最佳实践，优先采用。
- **核心架构做差异化**：微内核 + 插件架构是项目的核心创新点，插件注册、动态加载、权限控制等部分需要自研。
- **判断标准**：如果一个领域已经有行业标准或广泛认可的最佳实践，就研究它、采用它、适配它，而不是从零造轮子。

## MVP 迭代原则

**核心思想**：先做出来 → 跑起来 → 看效果 → 再调。
**宪法依据**：想成功先跑通 | 集中优势兵力各个歼灭敌人 | 实践是检验真理的唯一标准

- 不要一次性设计完美，优先验证全链路通畅
- 用真实数据快速试错，确认 API 和前端配合没问题后再优化
- 效果导向：代码写得再漂亮，用户不用也是白搭
- 每次迭代聚焦一个功能模块，方便归因

## 插件开发原则

**插件是 Arche 的唯一扩展方式**，所有新功能以插件形式实现。

- **目录结构**：一个插件一个目录，必须包含 `__init__.py`（入口+自注册）
- **内部分层**：简单插件可只放 `__init__.py`，复杂插件按 `models.py` / `routes.py` / `services.py` 分层
- **自注册**：插件在 `__init__.py` 中调用 `registry.register(name, plugin)` 完成注册
- **无需修改核心代码**：新增插件不改 `core/` 下任何文件

## 测试原则

**优先单元测试，减少命令行验证。**

- **单元测试有存档价值**：测试代码即文档，记录了当时测了什么、怎么测的、覆盖范围，可随时回溯
- **减少用户审批负担**：命令行验证每次都需要用户审批权限，单元测试只需 `uv run pytest` 一次通过
- **自动化回归保障**：后续重构或新增功能时，已有测试自动验证旧功能不被破坏
- **适用场景**：业务逻辑、API 接口、工具函数、数据校验等均可写单元测试
- **不适用场景**：UI 渲染、浏览器交互、端到端流程（这些仍需手动或 E2E 测试）
