## 2. 技术约束

### 2.1 数据库选型：PostgreSQL

**为什么不用 SQLite**：
- 爬虫高并发写入 + 云训练多实例并发 → SQLite 写锁冲突严重
- PostgreSQL 支持 JSON 列（爬虫清洗后的非结构化数据）
- PostgreSQL 全文检索（博客搜索）
- 本地 Docker 跑一个，零运维

**为什么不用 MySQL**：
- PostgreSQL 的 JSON 支持优于 MySQL 的 JSON 函数
- PostgreSQL 对复杂查询（窗口函数、CTE）支持更好
- 爬虫的 URL 去重、资源统计等场景更适合 PG

**连接方式**：本地 Docker 容器，`postgresql://veil:veil@localhost:5432/veil`

### 2.2 前端按需下发

**技术路径**：
1. Vite dynamic import → 按组件分割 chunk
2. 后端只返回当前等级允许的路由列表
3. 前端只 import() 这些路由对应的 chunk
4. 高等级 chunk 的 URL 不写到低等级页面 HTML 中
5. 所有 API 请求必须携带 token

**局限性**：技术能力强的人可以猜 URL，但防住 99% 场景。真正的安全靠后端 API 权限校验。

### 2.3 插件通信防循环

**方案**：
1. 声明式依赖（`requires` / `optional`）→ 启动前 DAG 拓扑排序验环
2. 运行时容器追踪调用栈（`_resolving` set）→ 兜底检测隐式循环
3. 禁止插件间直接 import → 代码规范约束

### 2.4 OSS 存储安全

**外部写入隔离**：
- 每个租户独立命名空间：`/tenant/{id}/`
- 路径安全检查：禁止 `../`、禁止绝对路径
- 认证：AK/SK 或 token

**个人文件冷热分层**：
- 触发策略：N 天未访问 → 推到阿里云
- 本地只留元数据 + 缓存指针
- 用户文件不参与冷热分层

## 3. 风险分析

| 编号 | 风险 | 影响 | 缓解 |
|------|------|------|------|
| R1 | GitHub 中转 GH 服务稳定性 | GH 挂了 → GitHub 代理全挂 | GH 需自行部署在可靠服务器上 |
| R2 | 爬虫被封禁 | 出口 IP 被封 → 爬不到数据 | 限速、轮换 UA、遵守 robots.txt |
| R3 | 阿里云 OSS 费用失控 | 冷热分层策略不当 → 大量上传 | 设置上传上限、手动确认 |
| R4 | 前端 chunk 泄露 | 高等级代码被猜测 URL | 后端 API 校验 + chunk 不暴露 URL |
| R5 | 插件循环依赖 | 启动死锁 | DAG 验环 + 运行时追踪 |
| R6 | 云训练实例泄漏 | SSH 失败 → 实例不销毁 | finally 兜底清理 |

## 4. 开发优先级建议

**阶段 1（基础设施）**：
1. PostgreSQL 容器搭建 + SQLAlchemy 集成
2. 核心容器（服务注册 + 依赖注入 + 循环依赖检测）
3. auth 插件（等级权限 + 用户认证）

**阶段 2（核心功能）**：
4. OSS 插件（本地存储 + 外部写入 + 租户隔离）
5. blog 插件（完整版：读/写/评论/点赞）

**阶段 3（增值功能）**：
6. asset_mgmt 插件（资产编目）
7. crawler 插件（爬虫 + 运维面板）

**阶段 4（高级功能）**：
8. github_proxy 插件（GitHub 1:1 + GH 中转）
9. cloud_integration 插件（云训练管线 + Streamlit 面板）
