# 漫游爬虫工厂设计文档

> 创建日期：2026-04-22

## 核心理念

不是传统"任务驱动型"爬虫，而是**24 小时常驻的漫游爬虫**，模拟真实用户的网络冲浪行为，自主发现、自主抓取、自主存储。

## 关键概念

### 1. 24 小时常驻
- 后台服务，启动后一直运行，除非手动停止
- 不需要用户手动创建任务，系统自主运转

### 2. 超级佛系模式
- 随机停留：1 秒到 10 分钟都有可能
- 随机行为：关页面、翻一翻、扔着不动——都可能
- 不强求：能拿到内容就拿，拿不到不纠结
- 反反爬效果极佳，行为特征极其自然

### 3. 数据爬虫工厂（流水线架构）
一条任务 = 一个 URL 的完整生命周期，依次经过各车间处理：

```
URL 进来 → 抓取 → 解析 → 分类 → 质检 → 存储
```

- 每条任务内部是**串行**的（类似 GitHub Actions step）
- 多条任务之间是**并行**的（全局控制并发数）
- 各车间是独立的内置处理单元，不可自定义替换

### 4. 自主任务并发
- 不需要用户创建任务，系统自主发现 URL 并加入队列
- 全局控制并发数（如同时最多 5 条任务在跑）
- 用户可以手动停止整个系统或塞入特定 URL

### 5. 种子来源与探嗅
- **初始种子**：从环境变量读取一批根路径（新浪、知乎、贴吧、CSDN、微博等）
- **自主发现**：从页面中提取链接，对每个新链接做轻量探嗅
- **探嗅**：轻量探测（不用浏览器，httpx 直接请求），判断页面类型，决定是否收为种子
- **反馈回路**：探嗅后有意义的链接加入种子池，形成自生长

### 6. 黑白名单
- **有意义**：只要页面有实质文字内容（非纯功能性页面），都收下（八方来客，啥都要）
- **无意义**：登录/注册页、纯广告页、纯导航页、报错页、纯功能性页面 → 扔掉 + 加入黑名单

## 各模块职责

### 种子管理器
- 从环境变量读初始种子
- 从探嗅/抓取结果中发现新种子
- 维护黑白名单

### URL 调度器 + 任务调配器（总调度）
- 管理待爬队列，控制全局并发数
- **按域名限流**：每个域名同时最多 2-3 个活跃请求
- 分配任务给流水线
- 域名独立：域名 A 跑域名 A 的，域名 B 跑域名 B 的，互不干扰

### 抓取车间
- 正式抓取：使用轻量 HTTP 请求获取页面内容，能拿则解析，拿不到则放弃或降级记录
- 不模拟浏览器行为，避免引入高维护成本、资源消耗和反爬触发面
- 探嗅模式：复用同一套轻量请求能力，快速判断是否值得继续处理

### 解析车间
- 提取正文、标题、链接列表
- 区分内容页与功能页

### 分类车间
- 判断页面类型（文章/帖子/导航/广告/功能页…）
- 分类打标

### 质检车间
- 排除纯功能性页面（登录、注册、404、纯广告）
- 其余全部放行

### 存储车间
- 一条任务产出一个文件
- 写入本地 OSS
- 文件过大时分片 + 压缩
- 文件格式：JSON 或 tar.gz
  - 文本/JSON 文件：直接写入 JSON 对象
  - 压缩文件：内部包含 manifest.json（元数据头）+ content.json（正文）+ raw.html（可选原始 HTML）

## 文件内容字段
- `url`：页面 URL
- `title`：页面标题
- `content`：正文内容
- `crawled_at`：抓取时间
- `source`：来源域名/页面
- `user_agent`：爬虫标识
- `content_type`：内容类型（文章/帖子/导航/…）
- `status_code`：HTTP 状态码
- `headers`：响应头信息

## 资源管理
- 探嗅/分类/质检不用浏览器，纯代码处理
- 只有正式抓取页面内容时才启动浏览器
- 根据服务器性能控制总并发数
- 浏览器实例复用，避免多实例内存爆炸

## OSS 存储链路
```
爬虫任务 → 生成文件 → 本地 OSS → OSS 自行决定何时同步到远程云
```
- 爬虫只负责写本地 OSS
- OSS 自己处理云同步
