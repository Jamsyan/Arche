# Veil 需求文档

> 日期：2026-04-21
> 状态：需求阶段

---

## 1. 项目定位

个人模块化平台，微内核架构。所有功能以可插拔插件形式扩展，前端按等级权限按需下发代码 chunk。

---

## 2. 整体目标

| 目标 | 说明 |
|------|------|
| 个人服务平台 | 博客 + 文件存储 + GitHub 镜像代理 |
| 模型训练管线 | 本地爬数据 → 清洗 → 云 GPU 训练 → 产物回传 |
| 资产统一管理 | 所有用户资产统一编目，不分散 |
| 等级权限体系 | 数字越小权限越高（0=最高），避免角色歧视 |
| 前端按需下发 | 不同等级用户只拿到对应 chunk，非该等级无任何前端代码可见 |

---

## 3. 插件清单

### 3.1 auth（认证与等级权限）

**核心职责**：用户认证、等级管理、权限校验。

**需求**：
- 等级数字体系：0 = 最高权限，数字越大权限越低
- 等级自动继承：Level N 自动拥有 Level N-1 的全部权限
- 权限校验拦截：所有 API 端点需携带认证身份，无身份直接拒绝
- 插件间权限查询：其他插件可通过核心容器查询当前用户等级

### 3.2 blog（完整博客）

**核心职责**：博客内容发布、浏览、互动。

**需求**：
- 浏览（读）：所有人开放，不需要登录（P5 游客即可）
- 写作：注册用户可以发帖，所有帖子必须经过审核
- 互动：评论、点赞，注册用户即可
- **博客独立质量等级体系**（发布者等级概念）：
  - 帖子内部独立一套等级，与全局 P 等级解耦
  - 升级条件：发帖数量多 + 质量高 + 浏览量高 + 审核通过率高
  - 降级条件：审核不通过直接降级
  - 具体升降级规则属于商业逻辑，暂不细化
- **审核流程**：
  1. 自动审核：敏感词过滤，过滤不通过的直接拦截
  2. P0 手动审核：随机抽审账号/帖子
  3. 风控触发：评论区举报/反馈 → 触发风控 → 先降流/限制预览 → P0 审核 → 不合格则删除/禁号，合格则恢复
- 所有博客内容（文章、评论、点赞记录）都是用户资产

### 3.3 oss（文件存储系统中转）

**核心职责**：文件存储管理，本地热存储 + 云端冷存储，外部写入支持。

**需求**：
- **用户上传文件**：帖子附件等，存储于本地，永久保留，不会推到云端
- **P1 独立存储**：P1 用户有本地独立 OSS 空间（约 10GB 上限），用于中转存储，不走阿里云，与用户存储隔离
- **个人私有文件**：用户自己的数据集、私人文件，本地热存储 + 阿里云 OSS 冷存储，冷热分层只涉及个人资产，与用户无关
- **冷热策略**：服务器磁盘仅 40GB 可用空间，量太大直接丢阿里云；时隔约 7 天未访问的文件推到云端
- **外部写入**：其他项目通过认证密钥写入文件，每个租户有独立命名空间隔离，不能越界访问
- **安全管理**：路径隔离（禁止 `../` 转义逃逸），访问控制（认证后写入）
- **云转接任务**：接收外部任务队列，从云端拉数据 / 推数据，支持模型训练数据集等场景
- **热冷策略**：个人文件的冷热分层由用户个人资产触发，不基于用户数据

### 3.4 github_proxy（GitHub 镜像代理）

**核心职责**：完整 GitHub 前端体验（去掉不必要的部分），通过中转代理访问 GitHub，解决国内访问慢的问题。

**需求**：
- **1:1 GitHub 前端**：保留仓库浏览、代码查看、PR/Issue 列表等核心功能，去掉 marketplace、企业功能等不必要的部分
- **中转方案待定**：考虑以下方案，需调研哪个可行/好用：
  - SSH 连接 GitHub 实现中转
  - GitHub CI/CD 作为中转通道
  - 其他可行方案
- **缓存层**：静态文件、响应体本地缓存，带 TTL
- **P1 及以上可用**：不对外公开
- **限流**：防止滥用

### 3.5 asset_mgmt（资产统一管理）

**核心职责**：所有用户资产的统一编目和查询。

**需求**：
- 博客文章 = 资产
- 博客评论 = 资产
- 点赞 = 资产
- OSS 上传的文件 = 资产
- GitHub 镜像的仓库 = 资产
- 数据集 = 资产
- 统一查询界面：可查看"某用户名下所有资产"，不分散在各处

### 3.6 crawler（爬虫与数据处理）

**核心职责**：24 小时不间断合法爬数据，采用轻量请求策略抓取公开内容，数据清洗后存储。

**需求**：
- **种子 URL 机制**：从预置的平台级 URL 出发（博客、贴吧、微博、抖音、快手、知乎、学术平台等知识/信息密集平台），向外衍生探秀
- **扩展逻辑**：
  1. 扫描到一篇文章/帖子，提取其中携带的所有链接
  2. 这些链接预存到 URL 队列
  3. 专门的任务逐个探测这些链接：有内容 → 抓取并清洗；无内容 → 链接报废
  4. 如果探测到的链接本身还包含子链接 → 认定为有意义的种子链接，加入种子库继续扩展
- **并行抓取**：轻量 HTTP 请求并发抓取；能拿则拿，拿不到则放弃或降级记录
- **数据清洗**：原始 HTML 清洗，去除标签/广告/脚本等特征，只保留结构化内容，不存原始数据
- **运维面板**：
  - 资源探测：当前扫描到多少新 URL、多少有效资源
  - 任务状态：多少个任务在跑、多少排队、多少完成
  - 流量占比：总出口带宽、各站点流量占比
  - 存储统计：清洗后存了多少条、磁盘占用、增长曲线
  - 清洗率：原始 HTML vs 清洗后数据量对比
- **仅 P0 可用**：不对任何用户等级开放

### 3.7 cloud_integration（云训练管线）

**核心职责**：云端 GPU 训练管线，Provider 抽象，远程训练 + 产物回传。**注意：服务器仅做本地调度和部署，实际训练资源在云端，因此支持多实例并发。**

**需求**：
- **Provider 抽象**：统一接口对接不同云厂商（智云星、阿里云等）
- **训练管线**：创建实例 → SSH 部署代码 → 远程训练 → 产物回传 → 校验 → 销毁实例
- **任务管理器**：记录任务状态、采集监控指标、采集费用数据、暂存训练产物、提供查询 API
- **多实例并发**：支持同时提交多个训练任务（如超参搜索场景），服务器只负责任务调度和监控，实际训练在云端 GPU
- **前端面板**（Streamlit）：任务列表、GPU 指标曲线、费用趋势、Loss 曲线
- **产物管理**：SHA256 校验、暂存中转、按需推 OSS / 回本地
- **任务生命周期**：PENDING → CREATING → RUNNING → DOWNLOADING → VERIFYING → COMPLETED / FAILED / CANCELLED
- **仅 P0 可用**：不对任何用户等级开放

## 4. 权限体系详细

### 4.1 全局等级（P0~P5）

| 等级 | 定位 | 可用服务 |
|------|------|----------|
| P0 | 最高（个人/Root） | 全部开放：爬虫管理、云训练、个人 OSS 冷热、GitHub 代理、博客审核 |
| P1 | 部分开放 | GitHub 代理可用、本地 OSS 独立空间存储（非阿里云）、博客高权限（可审核帖子）、爬虫不可用、云训练不可用 |
| P2 | 中级用户 | 博客发帖/评论/点赞、文件上传（用户级本地存储），GitHub 不可用 |
| P3 | 初级用户 | 博客基础操作（发帖需审核），文件上传受限 |
| P4 | 注册用户 | 仅博客评论/点赞，不可发帖 |
| P5 | 游客（默认） | 仅博客浏览，常规游客权限 |

### 4.2 博客独立质量等级体系

博客有独立的权限子体系，与全局等级解耦：

- **质量等级越高** = 发的东西质量越高 = 审核门槛越低
- **升级极难**：需要满足条件，甚至可能需要付费购买
- **降级极快**：一次审核不通过，直接降一级
- **审核策略**：高等级帖子审核门槛低但降级狠，低等级帖子审核门槛高

### 4.3 前端按需下发要求

- 非当前等级的用户，其对应的前端组件 chunk 不会下发
- F12 也看不到高等级组件的代码
- 所有 API 请求必须携带身份认证，无身份直接 401
- 高等级 API 端点的 URL 不写入低等级页面

## 5. 存储策略

### 5.1 数据存储分层

| 数据类型 | 存储位置 | 说明 |
|----------|----------|------|
| 用户信息、等级 | PostgreSQL | 关系查询、事务 |
| 博客文章、评论 | PostgreSQL | 全文检索、分页 |
| 爬虫任务、URL、清洗数据 | PostgreSQL（JSON 列） | 非结构化数据灵活 |
| 云训练任务状态、指标 | PostgreSQL | 时序指标存储 |
| 博客图片、用户上传文件 | 本地 OSS（磁盘） | 大文件，不走数据库 |
| P1 独立存储空间 | 本地 OSS（磁盘） | P1 专属，不走阿里云 |
| 个人私有数据集 | 本地 OSS → 阿里云冷 | 冷热分层（7 天 / 量大触发） |
| 模型 checkpoint | OSS → SFTP 回传暂存 | 超大文件 |

### 5.2 关键区分

- **OSS ≠ 数据库**：OSS 存文件/二进制 blob，不擅长关系查询、事务、全文检索。数据库存结构化数据，OSS 存大文件。两者互补。
- **用户数据 ≠ 个人数据**：用户上传的文件永远在本地，不会推到阿里云。只有用户自己的私有文件才会冷热分层。
- **磁盘限制**：服务器仅 40GB 可用空间。分配：
  - ~20GB：系统业务（PostgreSQL 数据 + 用户上传文件 + 爬虫清洗数据）
  - ~10GB：P1 独立中转空间
  - ~10GB：个人私有文件（触发冷热分层前的缓冲区）

### 5.3 冷热分层策略

- **触发条件**（满足任一）：
  - 文件超过 7 天未访问
  - 本地磁盘使用量接近阈值
- **范围**：仅个人私有文件，用户文件不参与
- **目标**：阿里云 OSS

## 6. 插件依赖关系

```
core (核心 + 配置 + 服务容器 + 数据库)
  ├── auth                  # 等级权限，无依赖
  ├── blog                  # 博客，optional: auth
  ├── oss                   # 文件存储，optional: auth
  ├── github_proxy          # GitHub 中转 + 前端，requires: auth, level ≥ 1
  ├── asset_mgmt            # 资产编目，optional: auth（扫描所有插件资产）
  ├── crawler               # 爬虫 + 运维面板，无依赖（个人使用）
  └── cloud_integration     # 云训练管线，无依赖（个人使用）
```

### 6.1 插件间通信

- 通过核心服务容器间接通信，不直接 import 其他插件
- 插件注册时声明 required（必须有）和 optional（优雅降级）依赖
- 核心检测到循环依赖时启动失败，明确报错
- 服务调用通过容器追踪调用栈，运行时检测到循环立即拒绝
